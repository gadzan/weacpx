# `/later` 一次性定时任务交互设计

日期：2026-05-23
状态：已确认，待实现计划

## 背景

weacpx 当前通过无图形界面的聊天频道驱动 agent 会话。定时任务能力表面上是“在未来某个时间，把一条消息发送到某个会话”，但它不是普通入站请求：触发时没有新的用户消息，也就没有新的 `reply` 回调。实现必须复用频道已有的路由和异步出站通道，把定时触发伪装成一轮来自创建来源聊天的普通 prompt。

内置微信智能体频道不是企业微信客服通道；它的关键限制是：用户每条输入后，智能体最多可通过微信回复 10 次。现有微信路由已经把这个限制拆成中间输出和最终输出配额，并会在中间输出到达 6 次时停止继续流式发送，等待最终消息汇总后再发给用户。定时任务不应重新实现 quota 逻辑，应复用现有路由。

第一版只做轻量的一次性定时任务，不做周期性任务，也不做图形化维护界面。

## 目标

- 用户可以在聊天频道中用简短命令创建一次性定时任务。
- 用户可以查看所有尚未执行的定时任务。
- 用户可以取消尚未执行的定时任务。
- 定时任务在触发时向创建时绑定的会话发送普通 prompt。
- 输入格式足够好记，同时严格限制可识别的时间表达，避免自然语言误判。
- 所有任务必须在 10 秒之后、7 天之内执行。
- 触发时的可见通知和 agent 回复复用创建来源聊天的现有路由与 quota 处理。

## 非目标

- 不支持周期性任务。
- 不使用 `remind` 语义；该能力不是提醒，而是计划在未来给会话发送消息。
- 不使用 `/plan`，避免和底层 agent mode 或内置命令心智冲突。
- 不支持延迟执行 weacpx slash command。
- 不支持复杂自然语言，例如“明早”“今晚”“周五晚上”“月底”。
- 不支持复杂权限模型或创建者所有权控制。
- 不支持错过任务后的补执行。
- 第一版不提供历史查询命令。
- 不重新实现微信 quota、分段、最终消息汇总或 pending final 逻辑。

## 命令表面

主命令：

```text
/later
```

短别名：

```text
/lt
```

命令形态：

```text
/later                 # 显示帮助
/later <time> <msg>    # 创建一次性定时任务
/later list            # 查看全局待执行任务
/later cancel <id>     # 取消待执行任务
```

`/lt` 与 `/later` 完全等价：

```text
/lt
/lt in 30m 检查 CI
/lt list
/lt cancel k8f2
```

## 核心语义

`/later` 的用户心智模型是：

> 在未来 7 天内的某个时间，把一条普通用户消息发送到创建时绑定的当前会话。

创建任务时必须存在当前会话。任务绑定的是创建时的当前会话，而不是触发时的当前会话。用户之后切换会话，不影响已创建任务的投递目标。

创建成功示例：

```text
已创建定时任务 #k8f2
执行时间：2026-05-23 周六 21:30
会话：backend-codex
内容：检查 CI
```

内容展示采用摘要策略：120 字以内完整显示，超过约 120 字则截断并追加省略号。

## 内容限制

定时任务只发送普通 prompt，不延迟执行 weacpx 命令。

如果 `<msg>` 以 `/` 开头，拒绝创建：

```text
定时任务只支持发送普通消息给当前会话，不支持延迟执行 weacpx 命令。

如果你想让 agent 讨论这段命令，请把它写成普通句子，例如：
/lt in 1h 请解释 /status 的作用
```

这样避免用户误以为 `/later` 可以定时执行 `/status`、`/cancel`、`/config set` 等管理命令。

## 时间语法

时间解析使用本机本地时区。创建成功时必须回显绝对日期和星期，避免相对表达产生误解。

所有任务必须满足：

- 执行时间 >= 当前时间 + 10 秒
- 执行时间 <= 当前时间 + 7 天

7 天是计划时间上限，不代表定时触发时一定有可用出站能力。触发时仍需要创建来源聊天有可用的频道路由；具体 quota、分段、最终消息汇总和延后发送由频道现有实现处理。

### 相对时间

英文半结构化：

```text
/lt in 10m 检查 CI
/lt in 2h 检查 CI
/lt in 1d 总结当前进展
```

中文半自然表达：

```text
/lt 10分钟后 检查 CI
/lt 2小时后 检查 CI
/lt 1天后 总结当前进展
```

支持单位：

```text
m / min / minute / minutes
h / hour / hours
d / day / days
分钟
小时
天
```

第一版只支持阿拉伯数字和整数单位；不支持 `一小时后`、`半小时后`、`1.5h`。

### 今天、明天、后天

```text
/lt at 21:30 继续处理
/lt today 21:30 继续处理
/lt tomorrow 09:00 看 PR

/lt 今天 21:30 继续处理
/lt 明天 09:00 看 PR
/lt 后天 14:30 继续 debug
```

`at 21:30` 等价于“今天 21:30”。如果今天该时间已经过去，拒绝，不自动改成明天：

```text
21:30 今天已经过去了。

请使用更明确的时间，例如：
/lt tomorrow 21:30 继续处理
/lt in 2h 继续处理
```

### 星期几

```text
/lt 周五 09:00 看 PR
/lt 星期五 09:00 看 PR
/lt fri 09:00 看 PR
/lt friday 09:00 看 PR
```

规则：

- 星期映射覆盖 7 天：
  - 中文：`周一` 到 `周日`，`周天`，`星期一` 到 `星期日`，`星期天`。
  - 英文：`mon` 到 `sun`，`monday` 到 `sunday`。
- 解析为未来 7 天内最近的那个同名星期几。
- 如果今天就是目标星期几且时间还没到，安排到今天。
- 如果今天就是目标星期几但时间已过，自动理解为下周同一天同一时间。
- 解析结果仍必须在 7 天内，否则拒绝。
- 成功时回显绝对日期。

## 不支持的时间表达

第一版不支持：

```text
明早
今晚
下午三点
周五晚上
下周一
月底
饭后
睡前
```

解析失败时统一引导：

```text
没能识别定时时间。

请使用这些格式之一：

1. 相对时间：
/lt in 2h 检查 CI
/lt 30分钟后 总结进展

2. 指定日期词：
/lt today 21:30 继续处理
/lt tomorrow 09:00 看 PR
/lt 明天 09:00 看 PR

3. 星期几：
/lt 周五 09:00 看 PR
/lt friday 09:00 看 PR

限制：只支持 10 秒之后、7 天之内的一次性定时任务。
```

超出 7 天：

```text
定时时间超出范围。目前只支持未来 7 天内的一次性任务。

例如：
/lt in 3d 检查
/lt 周五 09:00 看 PR
```

少于 10 秒：

```text
定时时间太近了。请至少设置到 10 秒之后。
```

缺少消息：

```text
缺少要发送给当前会话的消息。

例如：
/lt in 2h 检查 CI 是否通过
```

没有当前会话：

```text
当前没有会话，无法创建定时任务。

请先创建或切换会话，例如：
/ss codex --ws backend
/use backend-codex
```

## 列表

```text
/lt list
/later list
```

显示全局所有待执行定时任务，不限当前聊天或当前会话，并显示绑定会话。列表按 `execute_at` 升序排列，最近要执行的任务排在最前。

示例：

```text
待执行定时任务：

#k8f2  2026-05-23 周六 21:30  会话：backend-codex
检查 CI 是否恢复

#p91a  2026-05-24 周日 09:00  会话：frontend-claude
继续整理昨天的问题
```

没有待执行任务：

```text
当前没有待执行定时任务。
```

列表内容同样最多显示约 120 字摘要。

## 取消

```text
/lt cancel k8f2
/lt cancel #k8f2
/later cancel k8f2
```

成功：

```text
已取消定时任务 #k8f2。
```

找不到待执行任务：

```text
没有找到待执行定时任务 #k8f2。

可以用 /lt list 查看当前待执行任务。
```

如果任务已经执行、取消、错过或失败，第一版可统一按“没有找到待执行任务”处理，不引入历史查询心智。

权限模型：第一版按可信频道/个人工具假设。能执行 `/lt list` 的人，就能取消任意待执行任务。由于列表是全局的，多频道环境下会显示其它频道创建的待执行任务摘要和绑定会话名；这是 v1 的有意识取舍。`/later`、`/lt` 必须接入 `command-policy.ts`，继承现有管理命令的门控策略。

## 触发执行

到时间后先向创建来源聊天发送可见通知：

```text
执行定时任务 #k8f2
会话：backend-codex
内容：检查 CI 是否恢复
```

这条通知应通过创建来源聊天的现有出站路由发送。通知成功后，再把内容作为普通 prompt 投递到绑定会话。

如果没有可用频道路由或通知发送失败，任务标记为 `failed`，不再把 prompt 投递到 transport。失败原因只写日志；不能假设还能发送“失败通知”，因为失败原因可能正是出站不可用。

投递 prompt 时，agent 回复应推回创建任务时的 `chat_key` 对应聊天，复用现有 prompt 回复路径和 quota 处理，而不是直接调用底层 send API。对微信频道，这意味着继续使用既有的 `QuotaManager`、中间输出 6 次上限、最终消息汇总、pending final 等机制；`/later` 不单独判断“还剩几次回复”，也不绕过现有路由。

`executed` 的定义：prompt 已成功 dispatch 给 transport。该状态不等待 agent 完整回复，也不保证全部回复都已成功推回频道。

如果通知已成功发送，但后续 dispatch 失败，例如绑定会话不存在或 transport 不可用，尽量发送失败通知：

```text
定时任务 #k8f2 执行失败：当前会话不存在或 transport 不可用。
```

如果这条失败通知也因额度或通道问题发送失败，只记录日志。任务状态变为 `failed`。第一版不做重试。

调度执行是非交互的；prompt 进入 transport 后仍遵循绑定会话既有 mode 和权限策略。如果底层 agent 因严格权限模式等待审批，这是会话自身行为，不由 `/later` 额外处理。

## 调度机制与精度

第一版使用一个专门的轻量调度循环，而不是复用 daemon 30 秒心跳或 orchestration 60 秒进度轮询。

建议机制：

- 运行期每 1 到 5 秒扫描一次 `pending` 任务，找出 `execute_at <= now` 的任务并触发。
- daemon 启动时先加载状态并重扫任务：
  - `pending` 且 `execute_at < now`：按“错过任务”规则处理为 `missed`。
  - `pending` 且 `execute_at >= now`：等待后续调度循环触发。
  - 残留的 `triggering`：视为上次进程在触发中崩溃，标记为 `failed` 并记录日志，不自动重投。
- 调度 jitter 等于扫描间隔；如果扫描间隔为 5 秒，则“10 秒之后”的任务最晚约在计划时间后 5 秒内开始触发。
- 同一任务触发需要有进程内锁或状态转换保护，避免并发扫描重复投递。

`10 秒之后` 是创建下限，不是毫秒级精度承诺。它的目的主要是避免刚创建就触发造成竞态或用户体验混乱。

## 错过任务

如果程序关闭或 daemon 未运行，导致任务到点时没有执行：

- 下次启动发现 `executeAt < now` 的待执行任务。
- 不补执行。
- 标记为 `missed`。
- 默认 `/lt list` 不显示 `missed`。
- 第一版只记录日志，不提供 `/lt history`。

该规则避免几小时前或几天前的旧任务在重启后突然发送到会话。

状态持久化顺序要避免双投递：

- 在开始触发前先把任务从可被其它扫描器拾取的 `pending` 状态转出，或使用等价的执行锁。
- 通知成功、prompt 成功 dispatch 后，再持久化为 `executed`。
- 如果通知成功后、dispatch 前进程崩溃，重启时该任务不应再次自动投递；可以标记为 `failed` 并记录日志。v1 优先避免双投递，而不是保证崩溃后补投。

```text

/lt 周五 09:00 继续处理

/lt list

取消：
/lt cancel <id>

说明：
- 只支持一次性任务
- 时间必须在 10 秒之后、7 天之内
- 到点后会把消息发送到创建时绑定的会话
- 触发通知和 agent 回复复用现有频道路由；微信回复额度由现有路由控制

```

## 状态模型建议

实现时可新增独立状态域，例如 `scheduled_tasks`。每条记录至少包含：

- `id`：短 ID，列表和取消使用，展示时带 `#`。ID 使用适合移动端输入的短小写 base36/hex 字符串；生成时必须对待执行任务做碰撞检查，不能假设短 ID 永不冲突。取消时允许用户带 `#`，匹配建议不区分大小写并规范化为小写。
- `chat_key`：创建来源聊天。
- `account_id` / `reply_context_token` 或等价出站路由信息：用于触发时把通知和 agent 回复推回创建来源聊天；实际发送时优先使用 context-token store 里的最新 token。
- `session_alias`：创建时绑定的逻辑会话内部 alias。展示时再转换为当前频道可读的 display alias。
- `execute_at`：ISO 时间戳。
- `message`：待发送 prompt。
- `status`：`pending`、`triggering`、`executed`、`cancelled`、`missed`、`failed`。
- `created_at`：创建时间。
- 可选 `source_label`：频道/发送者摘要，用于未来增强全局列表。

第一版 `/lt list` 只展示 `pending`。

## 解析边界

`/later` 的时间语法是一个小型白名单解析器，不应混进现有扁平命令匹配逻辑。

建议实现边界：

- `parse-command.ts` 只负责识别 `/later` / `/lt`、`list`、`cancel`，以及把创建命令余下 token 交给专用解析器。
- 新增独立模块，例如 `parse-later-time.ts`，暴露 `parseLaterTime(tokens, now)`，返回 `{ executeAt, messageStartIndex }` 或结构化错误。
- 该模块独立单测覆盖：
  - 相对时间。
  - 今天/明天/后天。
  - 中英文星期几。
  - 过去时间、少于 10 秒、超过 7 天。
  - 不支持表达如 `半小时后`、`1.5h`。

## 设计约束

- 命令语法应在 `src/commands/parse-command.ts` 层保持可预测，避免把任意自然语言传给解析器猜测。
- 命令入口应加入 known command 列表，使频道能够识别 `/later` 和 `/lt`。
- 触发时应复用现有 router/session/transport 边界发送 prompt，不直接绕过 session 模型。
- 文档需要同步更新 `docs/commands.md` 和 `/help` 主题。
